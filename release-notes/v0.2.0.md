# Piramid v0.2.0

This is mainly a foundation release. The goal of this version is to make the project easier to run, easier to release, and easier to extend as this moves toward one binary RAG, vector search, embedding workflows, and inference-oriented systems work on consumer hardware.

This release includes a large internal architecture cleanup, improved CI and release automation, better documentation, and several storage/indexing reliability improvements.


Assisted by Codex:

#### Highlights

- Reworked the crate layout around clearer system boundaries: `server`, `services`, `runtime`, `collections`, `storage`, `index`, `compute`, `cache`, `search`, `embeddings`, `metrics`, `config`, `inference`, and `cluster`.
- Added a service layer between HTTP handlers and core collection/vector/search/embedding behavior.
- Split collection responsibilities into a dedicated collection domain layer, collection manager, cache manager, record store, checkpointing, compaction, duplicate detection, and operation modules.
- Moved distance and similarity kernels into `compute` so index/search paths can share cosine, dot-product, and Euclidean implementations cleanly.
- Added explicit future boundaries for `inference` and `cluster` so distributed inference and distributed systems work have a stable place to grow without leaking into storage or HTTP code.
- Added automated release publishing for GitHub releases, crates.io, GHCR Docker images, SBOM artifacts, and multi-platform binaries.

#### Architecture

- Introduced a cleaner separation between transport, orchestration, domain logic, persistence, indexing, search execution, and compute kernels.
- Added `runtime` for shared server state instead of keeping application state directly inside the HTTP layer.
- Added `services` for use-case orchestration so HTTP handlers stay focused on request/response work.
- Renamed manager-style components consistently so ownership and lifecycle responsibilities are easier to follow.
- Added `VectorReader` as the read abstraction used by index/search paths instead of coupling index code directly to collection internals.
- Added scaffold boundaries for future inference and cluster work without pretending those systems are implemented yet.

#### Storage, Collections, and Indexing

- Extracted `RecordStore` to own append-oriented record persistence and reduce collection-level persistence complexity.
- Added a collection append cursor so inserts no longer need to scan existing entries to infer the next write offset.
- Split collection operations into clearer read, write, metadata, and limit modules.
- Moved collection persistence, checkpoint, compaction, cache maintenance, duplicate detection, and search wrappers into explicit collection modules.
- Refactored Flat, HNSW, and IVF index paths around the shared index traits and vector reader boundary.
- Improved checkpoint behavior so ANN and global index persistence are coordinated at checkpoint time.
- Fixed duplicate WAL logging behavior and tightened mmap-disabled configuration handling.

#### Search, Embeddings, and Compute

- Split search execution into explicit engine, type, and utility modules.
- Kept search focused on filtering, scoring, sorting, truncation, and batch/query behavior instead of persistence or HTTP concerns.
- Refactored embedding provider code for OpenAI, local, and Ollama-style providers behind clearer provider/cache/retry/type boundaries.
- Preserved embedding cache behavior while making cache ownership clearer.
- Moved metric math out of observability naming and into `compute`, leaving `metrics` focused on telemetry and measurement.

#### API and CLI

- Removed the old interactive CLI flow in favor of explicit commands such as `piramid serve`.
- Kept the CLI animation behavior while making command execution more direct.
- Split server DTOs into API-specific request/response types for collections, vectors, search, embeddings, and admin flows.
- Simplified route setup and request helper boundaries.
- Kept health, readiness, version, config, vectors, collections, and embeddings handlers separated from service logic.

#### CI, Release, and Packaging

- Added a local `scripts/check.sh` workflow for formatting, linting, tests, and bench compilation checks.
- Added pre-push hook support so contributors can run the same checks before pushing.
- Updated Rust CI to run formatting, Clippy, tests, and bench validation.
- Updated website CI and website checks.
- Fixed Docker builds by copying the full build context needed by Cargo benches and manifests.
- Changed Docker publishing so images are not pushed on every normal push.
- Added a release workflow that runs checks, verifies the crate package, creates the version tag, builds release binaries, publishes crates.io, and pushes GHCR images from the same tagged source.
- Added Docker image metadata, versioned Docker tags, `latest`, and SHA tags.
- Added release-note files under `release-notes/` so release descriptions can be written in the repo and published automatically.

#### Documentation and Website

- Added a comprehensive architecture guide covering the current codebase boundaries and how the major systems relate.
- Updated the root README with a higher-level overview and architecture entry point.
- Added setup documentation.
- Added architecture and contribution content for storage, indexing, query execution, embeddings, roadmap direction, and project history.
- Removed the old dashboard app from the repo and shifted focus toward docs, website, API, CLI, and core engine work.

#### Package Notes

- The crates.io package now includes focused crate metadata, repository/homepage links, README metadata, keywords, categories, and a smaller package include set.
- Release notes, docs, website files, Docker config, scripts, and GitHub workflow files are intentionally excluded from the crates.io package.
- Docker images are published to GHCR with versioned tags and OCI metadata.

#### Upgrade Notes

- The preferred CLI flow is command-based. Use `piramid serve` instead of relying on the old interactive menu flow.
- The internal module layout changed substantially. Downstream code that imports internal modules directly may need import updates.
- Existing data-path and API behavior should remain focused on compatibility, but this release is primarily an architectural foundation release rather than a new feature-heavy database release.
